📄
Stage 1: File Ingestion & Hash Deduplication

Performs content-based file deduplication to prevent redundant processing at the earliest pipeline stage.

  • Calculates MD5 and SHA256 cryptographic hashes of input files
  • Checks if files with identical hashes already exist in the database
  • Prevents duplicate document processing before any costly parsing
  • Stores file hash information in the LanceDB file_hashes table
Key Modules:
  • file_hash_deduplication.py
  • clear_tables.py
↓
🔍
Stage 2: Document Parsing

Extracts text, tables, and image descriptions from documents using vision-enabled LLMs.

  • Converts PDF pages to high-resolution images
  • Uses vision models to extract text, tables, and interpret images
  • Generates document metadata such as title, summaries, and relevant hashtags
  • Creates a combined JSON output with all extracted information
  • Saves page images as encoded base64 strings for later reference
Key Modules:
  • stable_pdf_parsing.py
  • stable_excel_parsing.py
  • stable_docx_pptx_to_pdf.py
↓
🔄
Stage 3: Document-Level Deduplication

Checks for document-level duplicates and similar documents before saving to the database.

  • Uses vector similarity with high thresholds (0.99+) to identify duplicate pages
  • Detects updated versions of existing documents through partial content matching
  • Compares document metadata and embeddings to identify similarities
  • Classifies pages as new, duplicates, or updates to existing content
  • Provides detailed information about duplicate pages and their source documents
Key Modules:
  • pre_save_deduplication.py
  • pipeline_interaction.py
  • user_interaction.py
↓
💾
Stage 4: Document Storage in LanceDB

Stores parsed document data with embeddings in LanceDB for efficient retrieval.

  • Takes parsed JSON content and saves it to LanceDB documents table
  • Computes text embeddings for each page using a multilingual embedding model
  • Prepends necessary instruction contexts for proper embedding generation
  • Creates vector search indices for similarity-based retrieval
  • Saves both text content and image data (as base64) to LanceDB
Key Modules:
  • stable_save_to_lancedb.py
  • init_lancedb.py
↓
✂️
Stage 5: Content-Aware Chunking

Divides documents into semantic chunks while maintaining alignment with similar documents.

  • Uses existing chunks from similar documents as context for the LLM
  • Maintains chunk alignment between document versions
  • Intelligently identifies duplicate, updated, and new chunks
  • Creates semantically meaningful chunks (100-200 tokens each)
  • For new documents: Uses LLM-based intelligent chunking
  • For document updates: Aligns with previous version's chunk boundaries
Key Modules:
  • context_aware_chunking.py
  • integrated_chunking.py
  • agentic_chunking.py
↓
📋
Stage 6: Requirements Extraction

Extracts atomic requirements from document chunks and analyzes them.

  • Connects to LanceDB and retrieves document chunks
  • Uses LLMs to extract atomic requirements from each chunk's content
  • Determines if each requirement is software-related
  • Generates embeddings for each requirement
  • Maintains traceability between requirements and their source chunks
  • Reuses existing requirements for duplicate chunks
Key Modules:
  • extract_requirements.py
↓
🔍
Stage 7: Requirements Deduplication

Identifies and manages duplicate requirements across documents.

  • Checks newly extracted requirements against existing ones in the database
  • Uses both vector similarity and LLM-based semantic analysis
  • Applies high-precision duplicate detection to avoid redundant requirements
  • Uses a batch approach for efficient processing of multiple requirements
  • Provides confidence scores and rationale for duplicate detection
  • Maintains references between duplicates for traceability
Key Modules:
  • pre_save_reqs_deduplication.py
  • chunk_review.py

LanceDB Database Tables

📄 documents
Document pages with content, metadata, and embeddings for page-level similarity detection.
Key Fields:
pdf_identifier string
page_number integer
document_title string
md_content string
embedding vector(1024)
image_b64 string
🧩 document_chunks
Semantic chunks with alignment information for tracking relationships between document versions.
Key Fields:
chunk_id string
document_id string
chunk_text string
embedding vector(1024)
is_duplicate boolean
aligned_with_chunk_id string
is_updated boolean
📋 requirements
Extracted requirements with analysis results and traceability to source document chunks.
Key Fields:
requirement_id string
document_id string
title string
description string
is_software_related boolean
source_chunk_id string
embedding vector(1024)
is_duplicate boolean
🔐 file_hashes
File metadata and cryptographic hashes for file-level deduplication.
Key Fields:
file_path string
file_name string
file_size integer
md5_hash string
sha256_hash string
timestamp string

Requify Pipeline Controller manages flow between stages and can stop at any point for testing

Controller Steps: 1) Hash Check, 2) Parse, 3) Deduplication, 4) LanceDB Storage, 5) Chunking, 6) Requirements Extraction
